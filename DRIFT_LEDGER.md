## Resolved in this pass

### 0. Email configuration was explicit in code but brittle in practice

- Previous state:
  - Resend was configured with a hardcoded sender address at `notifications@leasemate.com.au`
  - live API verification showed the Resend account had no verified sending domains
  - `.env.example` did not explain how to run Resend cleanly before branded domain verification
- Fix:
  - added `RESEND_FROM_ADDRESS` and `RESEND_REPLY_TO` as explicit env-managed settings
  - changed email code to read sender settings from env instead of hardcoding them
  - documented the two supported sender modes in [LOCAL_BUILD_STRATEGY.md](/Users/carlg/Documents/AI-Coding/Local-leasemate/LOCAL_BUILD_STRATEGY.md) and [INTEGRATIONS.md](/Users/carlg/Documents/AI-Coding/Local-leasemate/INTEGRATIONS.md)
- Result:
  - the email integration is now explicit about the difference between immediate working delivery and branded domain delivery

## Remaining follow-up items

### 1. Full provider matching/orchestration remains partially documented but not fully implemented

- Current evidence:
  - invitation records exist in schema and are used throughout provider, billing, and automation flows
  - local UAT seeds provider invitations directly in [scripts/seed-test-data.mjs](/Users/carlg/Documents/AI-Coding/Local-leasemate/scripts/seed-test-data.mjs)
  - there is no current runtime path creating provider invitations from submitted requests
- Impact:
  - the repo has a strong post-invitation workflow but not a fully implemented auto-matching creation path
- Recommended follow-up:
  - decide whether to implement invitation creation/matching or explicitly document it as an operator/manual or future workflow

### 2. Ops metrics may still imply fields not returned by current aggregates

- Current evidence:
  - some ops UI cards expect richer request metrics than the current `getOpsDashboardStats()` payload obviously returns
- Impact:
  - this is more code/runtime drift than doc drift
- Recommended follow-up:
  - verify each ops dashboard stat against the server payload and either add the missing metric or simplify the UI

### 3. UAT seeded product pricing differs from current Stripe category fee registry

- Current evidence:
  - seeded provider products in the UAT guide show introduction fees like `$39.00` and `$29.00`
  - the Stripe category fee registry in [server/lib/stripe.ts](/Users/carlg/Documents/AI-Coding/Local-leasemate/server/lib/stripe.ts) uses category-level amounts of `25` and `15`
- Impact:
  - the provider catalog and Stripe checkout fee model may communicate different fee expectations
- Recommended follow-up:
  - reconcile whether provider product `introductionFee` or category-level Stripe registry is the canonical commercial contract
