# LeaseMate Launch Gate

Last updated: 2026-07-01

This is the single canonical end-state verification document for LeaseMate.

Use it to answer one question only:

- is this implementation change actually complete enough to treat as done?

This file exists to prevent duplicate checklist logic across:

- `LOCAL_BUILD_STRATEGY.md`
- `UAT-GUIDE.md`
- `INTEGRATIONS.md`
- thread handoff notes

Those documents still matter, but they do different jobs:

- `LOCAL_BUILD_STRATEGY.md`
  explains how to run and verify the app locally
- `UAT-GUIDE.md`
  gives role-based test flows
- `INTEGRATIONS.md`
  records integration truth and verification status
- `LAUNCH_GATE.md`
  is the final pass/fail gate

If any other document starts acting like a second final verification checklist, this file wins and the duplicate should be removed.

## 1. When To Use This Gate

Use this gate for any of these decisions:

- marking a LeaseMate implementation task complete
- deciding whether a workflow change is fully carried through
- deciding whether the product is ready for serious UAT
- deciding whether a deployment is launch-like enough to trust
- deciding whether docs, code, and workflow behavior are aligned

## 2. Gate Outcome Rules

LeaseMate can only be called **passed** when every required gate below is either:

- passed
- not applicable and explicitly stated

LeaseMate is **blocked** when any required gate fails, is unknown, or depends on an implied step that was not verified.

## 3. Non-Duplication Rules

To avoid multiple docs covering the same aspect:

- do not copy long step-by-step UAT flows into this file
- do not restate full local setup instructions here
- do not duplicate the integration registry here
- do summarize the exact pass/fail conditions that those docs must satisfy

## 4. Canonical Gate

### Gate A. Repo And Build Integrity

Required:

- canonical repo is `/Users/carlg/Documents/AI-Coding/Local-leasemate`
- schema changes and runtime code are in sync
- `npx pnpm@10.28.2 run check` passes
- `npx pnpm@10.28.2 run test` passes
- `npx pnpm@10.28.2 run build` passes
- the deployment entrypoint and runtime-specific import rules are valid for Vercel serverless execution

Fail if:

- local verification is skipped
- Vercel runtime can build but API entrypoint crashes on first request
- migration and schema work changed but were not verified together

### Gate B. Customer Workflow Completeness

Required:

- customer can sign in
- customer can create a draft request
- customer can add service items
- customer can submit the request
- customer request route resolves at `/requests/:id`
- customer-facing status text stays customer-safe and does not expose backend jargon

Fail if:

- request submission works but the customer cannot track the request
- links generated by backend flows point to dead routes
- internal statuses appear directly in customer views

### Gate C. Provider Onboarding And Approval Completeness

Required:

- provider can create a profile
- approval evaluation runs after provider signup, profile update, product add, product update, product delete, and operator re-check
- approval checks cover:
  - business name
  - valid 11-digit ABN
  - phone
  - contact email
  - suburb
  - at least 1 active product
- pending providers cannot accept opportunities
- pending providers cannot start payment
- eligible pending providers auto-activate when `approvalMode = auto`
- operator-held providers remain pending until operator action
- public provider signup route is reachable and does not blank-screen during auth warm-up

Fail if:

- onboarding UI exists but approval state is not updated
- approval state changes but provider UI does not reflect why
- provider can bypass approval through direct opportunity/payment actions

### Gate D. Invitation, Payment, And Release Completeness

Required:

- provider opportunity visibility follows invitation status
- address and customer details remain hidden until release
- accepted invitations trigger payment deadline enforcement
- Stripe checkout session creation works for approved providers
- webhook verification works with raw request bodies
- paid webhook flow persists the fee, triggers customer release, emits audit, and remains idempotent on duplicate delivery
- released opportunities expose customer details only after fee success and release success

Fail if:

- provider can see customer details before release
- payment success does not continue the workflow
- duplicate webhooks duplicate business state

### Gate E. Automation And Exceptions Completeness

Required:

- tracked automation tasks exist for:
  - provider approval evaluation
  - provider timeout
  - payment deadline
  - stale request cleanup
- exception generation still matches the documented codes and behavior
- operator surfaces can see and act on exception-driven work
- local fallback behavior and remote durable behavior are both accounted for

Fail if:

- a timed workflow step exists in docs but not in code
- an exception path is triggered in code but invisible to operators
- delayed job payload changes were made without updating producers and consumers

### Gate F. Ops Boundary Completeness

Required:

- operators can pause and reactivate providers
- operators can manually approve, hold, and re-check pending providers
- operator actions are auditable
- operator actions do not become a hidden replacement workflow for normal system behavior

Fail if:

- ops can mutate provider state without audit
- operator actions are required to compensate for a missing but supposedly automated mainline step, unless that gap is explicitly recorded

### Gate G. Integration Truth Completeness

Required:

- integration expectations match `INTEGRATIONS.md`
- preview environments are treated as non-canonical and secretless unless intentionally upgraded
- production-only integrations are not assumed to exist in preview
- sender policy, storage path, Stripe path, and scheduler path remain consistent with documented truth

Fail if:

- preview behavior is interpreted like production behavior
- secrets are assumed rather than verified
- code depends on undocumented integration behavior

### Gate H. Docs Alignment Completeness

Required:

- behavior changes update the relevant authority docs in the same change
- `README.md` points to the current canonical docs
- `THREAD_HANDOFF.md` reflects the current repo truth
- `AUTOMATION_PIPELINE.md` reflects implemented workflow, not aspirational workflow
- `UAT-GUIDE.md` reflects the visible UI and routes

Fail if:

- docs still describe removed or future behavior
- the same final verification responsibility is spread across multiple docs

## 5. Current Project-Level Known Gaps

These are important because they affect whether LeaseMate is fully launch-ready versus only implementation-complete for a narrower change.

### Gap 1. Live provider invitation creation is still not end-to-end implemented

Current truth:

- provider invitation records exist
- provider opportunity, payment, release, and automation flows exist
- local UAT still relies on seeded `provider_invitations`
- there is no fully implemented live matching engine creating invitations from submitted requests

Interpretation:

- LeaseMate has strong post-invitation orchestration
- LeaseMate does not yet have a fully verified launch-ready intake-to-live-matching path

Rule:

- no one should call the whole marketplace fully launch-ready until this gap is either implemented or explicitly accepted as a manual/operator-owned workflow

### Gap 2. Preview is verification-only, not a canonical integrated environment

Current truth:

- preview deployments require access sharing when protected
- preview is intentionally non-canonical for secrets and live integrations

Rule:

- preview success is useful evidence, but not proof that the integrated production workflow is ready

### Gap 3. Branded Resend delivery is not the canonical default yet

Current truth:

- immediate working sender mode exists
- branded sender mode depends on domain verification

Rule:

- do not claim branded email readiness until domain verification is complete

## 6. Pass Criteria By Decision Type

### For “task complete”

Pass when:

- the changed workflow gates pass
- no downstream step introduced by the change was skipped
- docs were updated where truth changed

### For “UAT complete”

Pass when:

- the relevant role flows in `UAT-GUIDE.md` pass
- the related gates above pass

### For “launch-like ready”

Pass when:

- all gates A through H pass
- the known gaps above are either resolved or explicitly accepted as out-of-scope platform limits

### For “fully launch-ready”

Pass only when:

- all gates A through H pass
- live invitation creation is no longer a seeded/manual gap
- production integrations are verified at the required level

## 7. Required Final Output For Any Completion Decision

When using this gate, the final decision should state:

- decision: passed or blocked
- scope checked
- gates passed
- gates failed or not applicable
- known gaps still open
- exact next action if blocked

If that output cannot be written clearly, the work is not actually complete.
